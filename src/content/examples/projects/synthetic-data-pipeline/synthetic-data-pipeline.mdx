<p class="section-label">Mission Overview</p>

Physical test drives are the gold standard for autonomous vehicle validation — and also the bottleneck. A single edge-case scenario (black ice, construction zone, sensor-obscuring precipitation) may require hundreds of hours of driving to accumulate enough labeled examples for reliable ML training. This pipeline inverts that ratio: define the scenario parameters once, press run, and generate millions of labeled sensor frames overnight.

The system combines a deterministic physics engine (NVIDIA PhysX 5) with a path-traced renderer (OptiX 8) and a sensor model layer that accurately reproduces LiDAR return intensity, radar Doppler shift, and camera lens distortion. Every output frame is automatically annotated with ground-truth bounding boxes, depth maps, semantic segmentation masks, and sensor noise metadata.

Determinism is the core invariant: given the same scenario YAML and random seed, every worker node on every hardware generation must produce bit-exact physics and sensor output. This is achieved by fixing PhysX solver substeps, using a seeded PRNG for all stochastic elements (weather particle positions, traffic agent decisions), and pinning the OptiX ray generation kernel launch bounds.

<div class="callout">
  <strong>Validation gate</strong> — Every new scenario type must pass a domain-gap test before entering the production catalog: a detector trained exclusively on synthetic frames from that scenario must reach ≥92% mAP on a matched real-world validation split. Scenarios that fail are sent back to the sensor model team for calibration.
</div>

<p class="section-label">Key Technical Decisions</p>

**Why determinism over raw throughput?** The ability to reproduce a specific training example after a model regression is invaluable for debugging. Without determinism, a model failure tied to a specific edge-case frame is unreproducible — you can't bisect the training data. Enforcing bit-exact replay cost approximately 8% throughput compared to a non-deterministic implementation, a trade-off considered worthwhile.

**Why Ray over Kubernetes Jobs?** Scenario batches vary widely in per-frame compute cost (clear day vs. heavy rain with 200 traffic agents). Kubernetes Job-based scheduling assigns fixed GPU slices and wastes capacity when easy scenarios finish early. Ray's dynamic task scheduling rebalances work mid-batch, improving GPU utilization from 71% to 94%.

**Why Parquet for the frame store?** Training pipelines need to slice datasets by scenario type, weather condition, or sensor modality. Parquet's columnar format with predicate pushdown lets the training data loader filter 4.2M frames to a specific subset in under 400 ms without reading the full dataset — critical for rapid ablation studies.

<p class="section-label">Lessons Learned</p>

The largest domain gap source was **rain droplet interaction with LiDAR beams**. Our initial model applied a uniform intensity attenuation factor per rainfall rate — but real LiDAR returns in rain show bimodal distributions: strong returns from nearby droplets and attenuated returns from distant objects. Fitting a two-component Gaussian mixture per rainfall intensity bin and sampling from it during render time closed the domain gap from 7.3% to 1.9% for rain scenarios specifically.

The second finding was that **traffic agent behavior diversity matters as much as sensor realism**. Early batches used a single IDM (Intelligent Driver Model) for all agents with randomized parameters. Detectors trained on this data generalized poorly to aggressive cut-in maneuvers. Adding a separate adversarial agent policy trained via a simple RL loop (SUMO + stable-baselines3) produced the rare near-miss scenarios that the real-world validation set contained.
