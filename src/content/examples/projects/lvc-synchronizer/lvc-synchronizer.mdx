<p class="section-label">Mission Overview</p>

LVC Synchronizer V1 is a real-time federation middleware layer that solves one of distributed simulation's hardest problems: keeping **Live**, **Virtual**, and **Constructive** entities coherent across heterogeneous network segments without sacrificing determinism.

The system acts as a time-space-position-information (TSPI) broker — consuming raw DIS PDUs from live range assets, normalizing them against HLA federate time, and re-publishing to constructive environments within a single network tick.

The coordinator uses a **Logical Time Factory (LTF)** pattern: each ingested event is tagged with a monotonic logical timestamp derived from GPS-disciplined NTP, then sorted into a priority queue before fan-out. This eliminates causal ordering violations that plague naïve bridge designs.

<div class="callout">
  <strong>Design constraint</strong> — All DIS-to-HLA translation must complete within a single 10ms simulation tick. Any PDU that cannot be translated in time is dead-reckoned forward using the last known state vector rather than dropped, ensuring constructive federates never observe entity discontinuities.
</div>

<p class="section-label">Key Technical Decisions</p>

**Why io_uring over epoll?** The DIS receiver must drain up to 40K UDP datagrams per second from a multicast group. Traditional epoll + recvmsg saturated a single core at ~22K PDU/s due to syscall overhead. Switching to io_uring's fixed-buffer registered I/O brought syscall cost near zero and allowed the same core to sustain 40K PDU/s with headroom.

**Why gRPC between ingestion and coordinator?** The ingestion and coordinator run as separate Kubernetes pods to allow independent scaling. gRPC over HTTP/2 with protobuf encoding gave us structured schema evolution (critical for DIS version interoperability) and built-in flow control without hand-rolling a wire protocol.

**Why KEDA for autoscaling?** Kubernetes' native HPA only scales on CPU and memory. PDU queue depth — the actual signal for coordinator saturation — is a custom metric. KEDA's ScaledObject CRD lets us define a queue-depth threshold directly in YAML, scaling coordinator replicas from 1 to 12 within ~20 seconds of a burst.

---

<p class="section-label">Lessons Learned</p>

The hardest problem was **coordinate frame alignment**. DIS uses geocentric ECEF; most HLA federation OMTs use a local flat-earth NED frame anchored to a range origin. Naïve matrix rotation introduced 0.6m of accumulated drift per 100km of entity travel due to floating-point cancellation. The fix was to carry a 64-bit `double` reference origin and operate in relative ECEF throughout the coordinator, only converting to NED at the HLA publish boundary.

The second surprise was **HLA time regulation overhead**. Enabling time-constrained/time-regulating mode on all federates caused coordinator throughput to drop 40% — RTI grant latency became the bottleneck. Solution: only the coordinator itself participates in HLA time management; downstream constructive federates receive best-effort attribute updates with a lookahead hint embedded in the FOM extension.
