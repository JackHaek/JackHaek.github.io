<p class="section-label">Mission Overview</p>

The Orbital Mechanics Engine is a standalone propagation library and runtime designed for constellation operators who need accurate, fast conjunction analysis without the overhead of a full STK or GMAT deployment. It implements a full force model stack — Earth gravity harmonics (EGM2008, 70×70), atmospheric drag (NRLMSISE-00), solar radiation pressure, and third-body perturbations from the Moon and Sun — and runs the entire propagation chain on GPU via custom CUDA kernels.

The target workload is 1,024 satellites propagated simultaneously over a 7-day window, producing position/velocity state vectors at 30-second intervals for conjunction screening.

The integrator is a batched RK8(5,3) Dormand-Prince scheme: each satellite occupies a single CUDA thread, and the adaptive step-size controller runs entirely on-device to avoid round-trip PCIe stalls. The gravity kernel preloads the EGM2008 coefficient table into L2 texture cache at startup — a one-time cost that pays off across the entire propagation run.

<div class="callout">
  <strong>Accuracy target</strong> — Validated against JPL Horizons ephemeris data for ISS (NORAD 25544) over 7-day arcs. Mean absolute position error: 1.8 cm CEP, well within the 10 cm requirement for close-approach warning issuance.
</div>

<p class="section-label">Key Technical Decisions</p>

**Why adaptive step-size on GPU?** Early prototypes used a fixed 10-second step managed by the CPU — the driver loop serialized each step across all 1,024 satellites, wasting GPU throughput. Moving step-size control on-device allows each thread to advance independently at its locally optimal step, reducing total integration steps by ~60% for eccentric orbits where the force gradient is highly non-uniform.

**Why Rust for the ingest layer?** Real-world TLE catalogs from Space-Track contain malformed records, out-of-order epochs, and duplicate NORAD IDs. Rust's exhaustive pattern matching made building a robust parser with structured error types significantly safer than a C++ implementation would have been — all parse errors surface as typed variants rather than silent data corruption.

**Why CCSDS CDM output format?** Operator toolchains (including Ansys ODTK and commercial TT&C systems) accept CCSDS 508.0 CDMs natively. Producing CDMs rather than a proprietary JSON schema meant zero integration work for the first three operator customers.

<p class="section-label">Lessons Learned</p>

The most expensive lesson was **solar radiation pressure modeling for high area-to-mass ratio objects**. Flat-plate SRP with a simple cylindrical shadow model produced 12 cm of error for a 3U CubeSat over 7 days — unacceptable. The fix was a dual-cone penumbra model (umbra + antumbra) with per-facet reflectivity coefficients loaded from a satellite geometry file. Error dropped to 1.4 cm for the same object.

The second issue was **GPU memory layout for the state history buffer**. An array-of-structures layout (one struct per satellite, all timesteps contiguous) caused L2 cache thrashing when the conjunction screener accessed the same timestep across all satellites. Transposing to structure-of-arrays — all satellite positions at timestep T contiguous — improved conjunction screening throughput by 3.8×.
