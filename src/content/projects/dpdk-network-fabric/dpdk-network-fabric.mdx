<p class="section-label">Mission Overview</p>

Standard simulation middleware — DIS over UDP, HLA over TCP — was designed for reliable delivery, not for microsecond latency. When a constructive simulation is ticking at 50 Hz and a live entity update arrives 30ms late, the dead-reckoning correction creates a visible discontinuity. This fabric eliminates that problem by moving all simulation message transport out of the Linux kernel and into DPDK poll-mode drivers running on dedicated CPU cores.

The result is a transport layer that behaves less like a network stack and more like a shared memory bus — with the difference that it operates across 64 geographically distributed simulation nodes.

The fabric exposes a topic-based publish/subscribe API (similar to DDS, without the spec compliance overhead). Topics map to Ethernet multicast groups; the TX core resolves topic names to group MAC addresses at publish time and uses DPDK's `rte_eth_tx_burst` to push frames directly to the NIC without touching the kernel. The RX core runs a continuous poll loop on the same PMD and dispatches received frames to registered subscriber callbacks via a lock-free MPMC queue.

<div class="callout">
  <strong>Key invariant</strong> — No dynamic memory allocation occurs on the TX or RX hot paths. All packet buffers are drawn from pre-allocated DPDK mempools at startup. This eliminates allocator jitter — a major source of latency outliers in kernel-stack-based transports.
</div>

<p class="section-label">Key Technical Decisions</p>

**Why poll-mode over interrupt-driven?** Interrupts introduce non-deterministic latency: the OS schedules the interrupt handler whenever it chooses, adding 100–800 µs of scheduling jitter. A dedicated poll-mode core burns 100% of one CPU core but delivers deterministic sub-100 µs response times — an acceptable trade for simulation applications where latency predictability matters more than CPU efficiency.

**Why ZeroSer over Cap'n Proto or FlatBuffers?** Both Cap'n Proto and FlatBuffers add a vtable or offset table that requires a pointer dereference per field access. ZeroSer's fixed-layout schema (generated from a DSL at compile time) places every field at a statically known byte offset — field access compiles to a single `mov` instruction. For DIS entity-state PDUs accessed 40,000 times per second, this difference is measurable.

**Why Ethernet multicast over IP multicast?** IP multicast requires IGMP group management and carries the overhead of IP header processing. For a closed simulation network where all nodes are on the same L2 segment, Ethernet-layer multicast with pre-provisioned IGMP snooping eliminates an entire protocol layer and shaves 3–5 µs from the forwarding path.

<p class="section-label">Lessons Learned</p>

The trickiest issue was **NUMA topology misalignment**. The DPDK mempool allocated packet buffers from node 0 memory, but the NIC's DMA engine was connected to node 1's PCIe root complex. Every packet buffer access caused a cross-NUMA memory hop (+120 ns). Fixing the mempool allocation to `rte_socket_id()` of the NIC port dropped p99 from 81 µs to 47 µs — a 42% improvement from a single configuration line.

The second issue was **multicast group exhaustion under topic proliferation**. Ethernet supports 2^23 multicast group addresses, but many managed switches limit IGMP snooping table size (the Arista 7050CX3 defaults to 2,048 groups). A simulation federation with fine-grained topics can exhaust this table, causing the switch to flood unknown multicast groups to all ports. Solution: a hierarchical topic namespace that bins topics into at most 512 multicast groups, with the fabric runtime doing a secondary software filter on the RX core.
